DC01: Active Directory Domain Services, DNS, DHCP centralizzato, Global Catalog

### AD

#### Nome dominio Active Directory
Il dominio Active Directory proposto per il progetto è:

- Nome DNS dominio: `corp.ilsogno.it`
- Nome NetBIOS: `ILSOGNO`
- Forest root domain: `corp.ilsogno.it`

### Domain Services
#### Obiettivo
DC01 sarà il primo Domain Controller del dominio `corp.ilsogno.it` e ospiterà il ruolo Active Directory Domain Services.

Active Directory Domain Services centralizza identità, autenticazione e autorizzazioni per utenti, computer e servizi aziendali. In questo modo gli utenti degli hotel accedono alle postazioni, ai file server, al gestionale e alle risorse interne usando credenziali personali e controllate centralmente.

#### Ruolo di DC01
  DC01 avrà il compito di:
  - creare e gestire il dominio `corp.ilsogno.it`;
  - autenticare utenti e computer tramite Active Directory;
  - gestire utenti, gruppi, computer e unità organizzative;
  - applicare Group Policy agli endpoint aziendali;
  - fornire i servizi di dominio necessari a file server, NPS, VPN e gestionale;
  - replicare Active Directory con DC02.

#### Struttura logica
La struttura AD sarà organizzata con unità organizzative separate per sedi, reparti e dispositivi. *Vedi documento per le GPO*

Questa struttura permette di applicare policy diverse in base al ruolo o alla sede, mantenendo una gestione centralizzata.
#### Autenticazione
Gli utenti accederanno con account personali. Anche sulle postazioni condivise, come i PC della reception, il login sarà sempre individuale.

Questo consente:
- tracciabilità delle attività;
- permessi assegnati per gruppo;
- revoca rapida degli accessi;
- gestione centralizzata delle password;
- applicazione automatica delle policy di sicurezza.

#### Ridondanza
  DC02 sarà promosso come Domain Controller aggiuntivo dello stesso dominio. In caso di manutenzione o indisponibilità di DC01, DC02 continuerà a fornire
  autenticazione, DNS e servizi di dominio.

#### Test di accettazione
- Un utente riesce ad accedere a un PC joined al dominio.
- Un PC riceve correttamente le Group Policy.
- Un utente accede solo alle cartelle autorizzate.
- La disabilitazione di un account blocca l'accesso alle risorse.
- Spegnimento DC01: login e risoluzione dominio continuano tramite DC02.


### DNS

#### Obiettivo
DC01 ospita il servizio DNS interno necessario al funzionamento di Active Directory. Il DNS permette ai client aziendali di trovare Domain Controller, servizi di dominio, file server, NPS, gestionale e risorse interne del CED.

Per i dispositivi aggiunti al dominio, il DNS interno non è opzionale: senza risoluzione DNS corretta, login al dominio, Group Policy, accesso a file server e autenticazione centralizzata diventano instabili.

#### Architettura
- DC01: DNS primario integrato con Active Directory.
- DC02: DNS ridondante integrato con Active Directory.
- Zone DNS: integrate in Active Directory e replicate tra DC01 e DC02.
- Client aziendali: ricevono DC01 e DC02 come DNS tramite DHCP.
- Reti guest e dispositivi non aziendali: usano resolver locale del firewall o DNS pubblici filtrati, non direttamente il DNS dei Domain Controller.

#### Zone DNS
La zona DNS interna principale sarà `corp.ilsogno.it`, corrispondente al dominio Active Directory.

La zona interna conterrà record per:
- Domain Controller: `dc01.corp.ilsogno.it` e `dc02.corp.ilsogno.it`.
- File server: `fs01.corp.ilsogno.it`.
- NPS/RADIUS: `nps01.corp.ilsogno.it`.
- Server gestionale: nome da definire in base al prodotto scelto.
- Eventuali servizi interni esposti solo alle reti aziendali.

#### Risoluzione DNS
```text
Client aziendale
    -> DNS DC01 / DC02
    -> record interno AD o CED
    -> se record non interno, inoltro verso resolver esterno
```

Le richieste per risorse interne vengono risolte da DC01/DC02. Le richieste Internet vengono inoltrate verso resolver esterni o verso il firewall.

#### Integrazione con DHCP
Gli scope DHCP delle reti aziendali distribuiranno come DNS primario e secondario gli indirizzi di DC01 e DC02.

Le reti ospiti, Smart TV, tablet non aggiunti al dominio e sale meeting useranno DNS separati, gestiti dal firewall locale. Questa separazione evita che dispositivi non fidati interroghino direttamente i Domain Controller.

#### Sicurezza e controllo
- Consentire query DNS ai Domain Controller solo dalle VLAN autorizzate.
- Bloccare accesso DNS diretto a DC01/DC02 da guest Wi-Fi e reti non fidate.
- Evitare record DNS pubblici per servizi interni.
- Documentare record statici e alias usati da applicazioni, file server e gestionale.

#### Test di accettazione
- Client aggiunto al dominio risolve DC01, DC02 e il dominio AD `corp.ilsogno.it`.
- Client staff accede a file server e gestionale usando nomi DNS, non indirizzi IP.
- Client guest naviga su Internet ma non può interrogare direttamente il DNS dei Domain Controller.
- Spegnimento DC01: la risoluzione interna continua tramite DC02.

### DHCP Centralizzato

#### Obiettivo
DC01 eroga gli indirizzi IP per le VLAN delle sedi tramite Windows DHCP Server. La gestione centralizzata consente controllo unico degli scope, tracciabilità dei lease e configurazione coerente tra hotel.

DC01 quindi deve ospitare il ruolo Windows DHCP Server, autorizzato in Active Directory. Ogni VLAN/sottorete avrà uno scope dedicato su DC01. DC02 sarà configurato in failover DHCP con DC01.

*Il DHCP è centrale, ma il gateway resta locale*. Quindi un PC a La Meraviglia riceve l’IP da DC01, ma come default gateway riceve l’interfaccia VLAN del pfSense locale. Questo mantiene coerente il local breakout già deciso per guest, Smart TV, tablet e sale meeting.

#### L'eccezione
> Per le mini-case: DHCP locale sul router della mini-casa, ma configurazione standardizzata e documentata centralmente. Così se il tunnel WireGuard cade, la rete interna della mini-casa continua almeno a funzionare localmente.

#### Flow
```text
Client VLAN hotel
    -> gateway VLAN / pfSense locale
    -> DHCP relay
    -> VPN verso CED
    -> DC01 / DC02 DHCP
    -> lease restituito al client
```

#### Architettura
- DC01: DHCP primario
- DC02: DHCP failover
- pfSense sedi: DHCP relay per VLAN
- Gateway VLAN: locale alla sede
- DNS distribuito via DHCP: DC01 e DC02 per reti aziendali

#### Ambiti DHCP
Tabella scope per sede, VLAN, subnet, gateway, pool, esclusioni, DNS, lease.

#### DHCP Relay
La Meraviglia e L’Incanto inoltrano le richieste DHCP al CED tramite VPN IPsec. Il Sogno usa relay locale verso DC01/DC02.

Le mini-case usano DHCP locale sul router, con configurazione standardizzata e documentata centralmente.

#### Failover
DC02 replica gli scope DHCP di DC01. In caso di indisponibilità di DC01, DC02 continua a rilasciare lease.

#### Sicurezza e controllo
- DHCP Snooping sugli switch, dove supportato.
- Server DHCP autorizzati in Active Directory.
- Logging dei lease e degli eventi DHCP.
- Backup periodico della configurazione DHCP.
- Monitoraggio utilizzo scope.

#### Test di accettazione
- Client staff riceve IP, gateway e DNS corretti.
- Client guest riceve IP ma non raggiunge reti interne.
- Client AD risolve il dominio interno.
- Spegnimento DC01: lease erogato da DC02.
- VPN sede remota attiva: relay funzionante.
- VPN sede remota down: comportamento documentato.

### Global Catalog

DC01 sarà configurato come Global Catalog Server. Il Global Catalog mantiene un indice degli oggetti principali di Active Directory e consente ricerche rapide di utenti, gruppi, computer e risorse.

Nel progetto *Axion* il Global Catalog è utile per garantire autenticazione efficiente degli utenti, risoluzione dei gruppi e disponibilità dei servizi di dominio per tutte le sedi collegate al CED.

Anche DC02 sarà configurato come Global Catalog Server, così da garantire continuità operativa in caso di manutenzione o indisponibilità di DC01.
